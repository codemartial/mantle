01 Product brief

A small, sharp tool for a single, common workflow — not a full DAM. Every decision in this doc bends toward keeping that surface narrow.

Target user

Serious enthusiast / travel photographer with a working library in the tens of thousands. Lightroom-curious but doesn't want a catalog. Travels with a laptop, edits in batches per trip, cares about IPTC captions / keywords / geo more than rating or culling.

Job-to-be-done

"I just got back from a trip. I have a folder of photos. I want to tag, caption and locate them before posting / archiving — without committing to a DAM."

In scope

  • Read EXIF, IPTC, XMP from any image format
  • Edit headline, caption, keywords, date/time/TZ, lat/lon
  • Auto-save to XMP sidecar, never modify originals
  • Single-image and multi-image (batch) editing
  • Filter by date, camera/lens, keyword, missing fields

Out of scope

  • Catalog / library across folders
  • Rating, color labels, flags, picks/rejects
  • File rename, move, copy, delete
  • Image rendering / development / export
  • Cloud sync, collaborative editing
  • Histogram, sharpening, color tools

Platform & scale

OS
macOS 14+ (Sonoma) recommended. Sandboxed.
Window
Fits 940 × 500 logical pixels minimum (FHD HiDPI with menubar margin). Resizable; preview area scales.
Library scale
Tens of thousands of images per folder; one folder at a time.
Performance budget
Folder open < 800ms first paint. Thumbnail grid scroll = 60fps. Save-to-sidecar < 50ms.
Distribution
TBD — see open questions.

02 Concept

Three-pane layout

Browser (left) · Preview + caption editor (center) · Metadata panel (right). Borrowed from photo apps users already know — minimum learning curve.

Sidecar-first save model

All edits land in a sibling .xmp file. Originals (especially RAW) are never modified. Repeated edits update the sidecar in place.

Auto-save

Edits commit on selection change and on app quit. Status bar shows "All changes saved" or briefly "Auto-saved". No manual Save button.

Master-driven batch

Multi-select promotes the first selected image to master. Editors prefill from master; edits apply to the selection. Keyword chips show scope: solid = in all, dashed = in some.

03 Screens & states

Three primary states are mocked; six secondary states are enumerated in §12.

3.1 Single-select (default)

Anatomy

  1. Titlebar — traffic-light dots, folder picker (the folder-name button at the left also opens a folder tree), sort, grid/list toggle, filter input. Auto-save status lives in the bottom status bar, not the titlebar.
  2. Browser — scrollable grid (or list) of thumbnails. Density tweakable. Numbered badge for batch order in multi-select.
  3. Preview — fit-to-frame photo with corner crop marks; floating zoom toolbar (Fit / 1:1 / rotate); zoom readout bottom-right.
  4. Caption editor — headline (1 line, 40-glyph counter) + caption (multi-line).
  5. Metadata panel (right) — Camera EXIF · Captured (editable date/TZ) · File · Location (map + lat/lon) · Keywords.
  6. Status bar — item count, current filename + size, save state, sidecar tag, color profile, app version.

3.2 Multi-select / batch edit

Master-first rules

  • Master = batchOrder[0], the earliest selected image. Badged M in the browser.
  • Headline prefilled with master's value. Editing applies to all selected images on auto-save.
  • Caption has two modes:
    • Replace — textarea prefilled with master's caption; edits replace caption on all.
    • Append — master's caption shown locked/greyed at the top of the field; the editable area below is appended to every image's existing caption.
  • Keywords render as a union with two scopes:
    • all Solid chip with a filled accent dot — the keyword exists on every selected image.
    • some Dashed chip with a muted dot — the keyword exists on at least one but not all.
    × removes a keyword from all. Right-click opens a menu: "Add to all" (promotes a some-chip), "Remove from all". Typing into the input and pressing , or ↵ adds the new keyword to every image.
  • Date — master's date editable; a separate Shift all by Δh Δm control offsets every selected image's timestamp while preserving relative order.
  • No-op guarantee — if before-data equals after-data for an image, that image's sidecar is not rewritten.

5.3 Auto-save triggers

Edits commit to the sidecar when any of:

  1. Selection changes (current image's dirty edits flushed)
  2. Batch mode entered or exited (pending batch draft applied)
  3. App is hidden / loses focus (proxy for quit)
  4. App quits
Guarantee: if computed-after equals stored-before, nothing is written. The sidecar's xmp:MetadataDate only advances on actual change.

06 Data model

6.1 In-memory image record

// Each photo has a stable id (file path), an immutable EXIF/file readout,
// and a mutable edits record. Edits flow through the EditStore and persist
// via the SidecarIO module.

struct ImageRecord {
    let id: String              // absolute file path
    let file: URL
    let fmt: String             // "RAW + JPEG", "JPEG", …
    let dim: CGSize             // pixel dimensions
    let size: Int64             // bytes on disk
    let colorProfile: String    // "Display P3", "sRGB", …

    // Read-only EXIF
    let camera: String
    let lens: String
    let shutter: String
    let aperture: String
    let iso: Int
    let focal: String
    let originalCaptureDate: Date

    // Read from XMP/IPTC; mutable
    var headline: String        // dc:title  (max 40 visual glyphs)
    var caption: String         // dc:description
    var keywords: [String]      // dc:subject, ordered set
    var captureDate: Date       // overrides EXIF if user edits
    var timezone: TZRule        // .auto(fromGPS) | .fixed(offset, label)
    var latitude: Double?       // exif:GPSLatitude
    var longitude: Double?      // exif:GPSLongitude
}

enum TZRule {
    case auto                   // resolved from GPS each render
    case fixed(offsetMinutes: Int, label: String)
}

6.2 Edit store

Holds the live, in-memory edits keyed by image id. Acts as the single source of truth for the UI; the SidecarIO module flushes from here.

final class EditStore: ObservableObject {
    @Published private(set) var images: [String: ImageRecord]
    private var dirty: Set<String> = []
    private var lastSaved: [String: ImageRecord]   // for no-op check

    func update(_ id: String, _ patch: (inout ImageRecord) -> Void)
    func flush() async throws         // called on selection change / hide / quit
    func isDirty(_ id: String) -> Bool
}

6.3 Batch draft

struct BatchDraft {
    var headline: String        // prefilled from master
    var captionMode: CaptionMode = .replace
    var captionReplace: String  // prefilled from master
    var captionAppend: String   // empty
    var addKeywords: [String]   // staged adds
    var removeKeywords: [String]
    var dateShift: DateComponents  // hours, minutes
    var masterDate: Date?
    var masterTz: TZRule?
}

enum CaptionMode { case replace, append }

Keywords in batch operate directly on the underlying images (no staging): adding writes to all, X removes from all, "Add to all" promotes a some-chip. Headline, caption and date are staged in BatchDraft and committed on auto-save.

6.4 Undo

Per-image edits push to a single linear undo stack (NSUndoManager with a custom group per save boundary). Multi-select edits group into a single undoable transaction.

Date round-trip: EXIF and IPTC have separate date fields without offsets; XMP carries the offset. The store always normalizes to a Date + offset and writes back to XMP only. EXIF date fields are not kept in sync — the sidecar is canonical.

08 Sidecar I/O

One {basename}.xmp per original. Adobe-flavoured XMP packet, UTF-8, LF line endings, no BOM. Atomic write via temp-file + rename.

8.2 Atomic write

  1. Build packet in memory.
  2. Write to .{basename}.xmp.tmp.{pid} in the same directory.
  3. fsync the tempfile.
  4. rename(2) to {basename}.xmp.
  5. If a previous sidecar existed, its mtime is preserved in the file's resource fork as a "previous version" hint (deferred).

8.3 Concurrent-edit policy (field-domain merge)

Metadater is single-app, single-window. We do not support concurrent editing, but the file format means another tool can still touch the sidecar between two of our writes. The safe behaviour:

  • Metadater owns: dc:title, dc:description, dc:subject, photoshop:DateCreated + offset, exif:GPSLatitude, exif:GPSLongitude, xmp:MetadataDate.
  • Every other property in the sidecar is passthrough: read it, preserve it, write it back unchanged. Ratings, color labels, develop settings, hierarchical keywords, GPS altitude — all left alone.
  • On flush, the writer re-reads the sidecar from disk, swaps in only the owned fields from the in-memory record, then atomically writes the merged document. We do not cache a "before" snapshot for diffing.
  • If both Metadater and another tool happened to edit the same owned field between flushes: last-writer-wins on that field. Fields outside the overlap survive both.

No prompts. No conflict UI. No 3-way diff.

09 Architecture (recommended)

9.1 Stack

UI
SwiftUI + AppKit bridges where needed (NSCollectionView for the browser grid at 10k+ scale; SwiftUI for everything else).
Image reading
ImageIO for JPEG/HEIC/PNG/TIFF and most RAW EXIF. ImageIO will NOT round-trip arbitrary XMP losslessly.
XMP read/write
Vendor ExifTool as an embedded binary, called per file via NSTask. Fallback path: a thin pure-Swift XMP parser/writer for the subset of fields we touch. ExifTool gives correctness; the Swift writer keeps latency low for hot path. Pick one — see open questions.
Maps
MapKit. For "offline maps only" requirement, fall back to a pre-rendered tile cache or a raster style asset; MKMapView supports custom overlays.
Thumbnails
QuickLookThumbnailing (QLThumbnailGenerator) with on-disk cache keyed by (path, mtime, size, resolution).
Folder watch
FSEvents.
Persistence
Only sidecars on disk. Optional SQLite index for fast filter/search across the open folder. Per-folder, rebuildable, not the source of truth.
Undo
NSUndoManager. Group by save boundary.

9.3 Threading

  • UI on main thread.
  • EditStore mutations dispatched on a serial actor.
  • SidecarIO writes on a background queue, batched by debouncer (200ms) to coalesce rapid selection switches.
  • Thumbnail generation off-main with a concurrency cap of ProcessInfo.processInfo.activeProcessorCount.
  • LibraryIndex initial scan off-main; UI shows progressive thumb fill.

13 Open questions / decisions deferred

  1. Theme default — follow system or remember last? Recommend follow system, with manual override in Preferences.
  2. XMP writer — vendor ExifTool (1 MB binary, slow first call, perfect correctness) vs hand-roll a Swift XMP writer for the 6 fields we touch (smaller, faster, more brittle). Recommend hand-roll for write, ExifTool only for read of unusual files.
  3. Sidecar naming — foo.NEF.xmp vs foo.xmp? Adobe / Lightroom use foo.xmp (extension replaced) and tools have historically disagreed. Decision: use foo.xmp (Adobe-compatible) but read both.
  4. External change reconciliation — resolved: field-domain merge (see §8.3). No prompts; Metadater rewrites only its owned fields, passes everything else through.
  5. RAW + JPEG pairs — treat as a single browser entry or two? Recommend single, with both files getting the same sidecar.
  6. Distribution — direct (Sparkle updater) vs App Store (sandbox restrictions limit FSEvents)? Recommend direct.
  7. HEIC live photos — preserve the .MOV pair? Yes (deferred to milestone 2).
  8. Pricing — paid one-shot? subscription? free with paid pro tier? Out of design scope, flagging.

14.3 Glossary

Master image
The first image selected in batch order — drives prefill values for headline, caption, date, etc.
Sidecar
The .xmp file written alongside an original to hold edits. Never modifies the original file.
Some / all scope
For batch keywords — "in all" = keyword exists on every selected image; "in some" = exists on at least one but not all.
Dirty
An image with edits not yet flushed to its sidecar.
Auto-save flash
The brief green "Auto-saved" indicator in the status bar that confirms a flush happened.
